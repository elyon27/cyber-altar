**Diagnosis**

- The build fails during type-checking because `Props` defined in `src/app/prayer-wall/page.tsx` does not satisfy Next.js’ `PageProps` constraint. The error is reported in the build logs at [lines 79-82](#L79-L82).

**Cause**

- `generateMetadata` (and possibly the page component) declares a `Props` type whose `params` property is a plain object (`{ slug: string }`). Next.js’ `generateMetadata` signature requires its first argument to extend `PageProps`, where `params` is defined differently (it’s optional and, for the current Next.js version, treated as a `Promise`). Because the custom `Props` type doesn’t match that shape, TypeScript raises the constraint error.

**Solution**

- Update the typing so the arguments passed to `generateMetadata` (and any related functions) genuinely satisfy `PageProps`. If you don’t need route parameters on `/prayer-wall`, drop the custom `Props` entirely. Otherwise, import `PageProps` from `next` and ensure your type matches it. For example:

  ```tsx
  // src/app/prayer-wall/page.tsx
  import type { Metadata, ResolvingMetadata, PageProps } from 'next';

  type PrayerWallPageProps = PageProps; // no slug on this route

  export async function generateMetadata(
    _props: PrayerWallPageProps,
    _parent: ResolvingMetadata
  ): Promise<Metadata> {
    return {
      title: 'Prayer Wall',
      // …rest of metadata
    };
  }

  export default function PrayerWallPage() {
    // …component code
  }
  ```

  If you really do need a slug, adjust to the shape Next expects:

  ```tsx
  type PrayerWallPageProps = PageProps<Promise<{ slug: string }>>;

  export async function generateMetadata(
    { params }: PrayerWallPageProps,
  ): Promise<Metadata> {
    const { slug } = await params;
    // …use slug
  }
  ```

  Modify the component accordingly (await `params` before using it). After aligning the types with `PageProps`, rerun the build and it should pass.
